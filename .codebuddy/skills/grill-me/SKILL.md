# 方案深度审查

## 概述

此技能用于对用户的设计方案或计划进行系统性、深度的审查性访谈。通过沿决策树逐一追问，确保方案的每个方面都被充分考虑，最终达成清晰、可执行的共识。

## 核心流程

1. **逐问题追问**：每次只问一个问题，不一次抛出多个问题，确保每个点都被充分讨论。

2. **遍历决策树**：沿设计方案的每个分支深入，逐个解决决策之间的依赖关系。不浮于表面，将每个模糊点掰开揉碎。

3. **主动给出推荐方案**：对每个问题，不仅要提问，还要给出 AI 推荐的答案或方案，帮助用户决策。

4. **代码优先**：如果某个问题可以通过探索现有代码库来回答，优先查看代码而非凭经验猜测。

## 对话模式

- 每轮对话只提一个问题
- 问题应聚焦于方案的关键决策点和潜在风险
- 当用户回答后，根据答案继续深入追问相关依赖项
- 保持对话持续直到：
  - 所有关键决策分支都已覆盖
  - 双方对方案达成一致理解
  - 用户明确表示可以结束

## 追问维度

根据方案类型，从以下维度进行追问：

| 维度 | 关注点 |
|------|--------|
| 边界条件 | 空状态、加载态、错误态、极端输入如何处理？ |
| 数据流向 | 数据从哪里来？谁负责数据？如何传递和更新？ |
| 状态管理 | 哪些是本地状态？哪些需要持久化？状态冲突时以谁为准？ |
| 兼容性 | 与现有代码的集成方式是什么？是否会破坏已有功能？ |
| 安全与性能 | 是否有 XSS/CSRF 风险？大量数据时的性能如何？ |
| 依赖关系 | 依赖哪些外部模块？旧逻辑是否可以复用？ |
| 用户交互 | 用户的操作路径是什么？有无快速操作或键盘快捷键？ |

## 结束条件

当满足以下任一条件时，结束审查并给出总结：

- 所有分支的决策依赖都已解决
- 用户主动表示"可以了"、"没问题了"、"结束"
- 方案已经达到可执行的清晰度

结束时应总结：
1. 已确认的决策点
2. 仍存在的风险或待定事项
3. 建议的下一步行动

## 示例对话

```
用户：我想为论坛帖子添加一个"收藏"功能，grill me！

AI：好，我们从数据层面开始。收藏数据应该存在哪里——本地 localStorage 还是同步到服务端？我推荐同步到服务端，这样可以跨设备使用，但需要先确认 API 是否已存在。

用户：同步到服务端，API 已经有一个 /favorites 端点。

AI：那么接下来需要考虑：帖子被删除后，收藏记录如何处理？我推荐前端在渲染收藏列表时做空值过滤，同时后端在删除帖子时也可以考虑级联清理。
...（继续深入追问）
```
